## Компоненты

### 1. Точка входа ([`main.go`](../main.go))

Запускает корневую команду Cobra. Никакой дополнительной логики не содержит.

### 2. CLI-команды ([`cmd/`](../cmd))

Используется библиотека [`spf13/cobra`](https://github.com/spf13/cobra). Определены три команды:

- **`root`** ([`cmd/root.go`](../cmd/root.go)) — корневая команда, точка входа для всех подкоманд.
- **`serve`** ([`cmd/serve.go`](../cmd/serve.go)) — запускает DNS-сервер и веб-интерфейс. Читает конфигурацию из переменных окружения, инициализирует БД, репозитории, кэш, DNS-хендлер и HTTP-сервер. Ожидает сигналы SIGINT/SIGTERM для graceful shutdown.
- **`migrate`** ([`cmd/migrate.go`](../cmd/migrate.go)) — управление миграциями БД через [`golang-migrate/migrate`](https://github.com/golang-migrate/migrate). Подкоманды: `up` (применить), `down` (откатить).

### 3. Слой данных ([`internal/repository/`](../internal/repository))

Реализует паттерн **Repository** для работы с SQLite.

- **`ZoneStorage`** ([`zones.go`](../internal/repository/zones.go)) — CRUD-операции над таблицей `zones`. Реализует интерфейс `ZoneRepository`.
- **`RecordStorage`** ([`records.go`](../internal/repository/records.go)) — CRUD-операции над таблицей `records`. Реализует интерфейс `RecordRepository`.

Интерфейсы (`ZoneRepository`, `RecordRepository`) позволяют при необходимости заменить реализацию хранения (например, на PostgreSQL) без изменения вызывающего кода.

### 4. DNS-сервер ([`internal/server/`](../internal/server))

- **`Handler`** ([`handler.go`](../internal/server/handler.go)) — основной DNS-хендлер, реализует логику обработки входящих DNS-запросов:
  1. **Проверка кэша** — если ответ есть в in-memory кэше и не истёк, возвращается сразу.
  2. **Проверка локальной зоны** — итеративный поиск зоны: от полного имени домена к родительским зонам (например, для `sub.example.com` проверяются `sub.example.com.`, `example.com.`, `com.`). Если зона найдена, выполняется локальный резолвинг.
  3. **Локальный резолвинг** — поиск записей в БД по домену и типу. Если запрошенный тип не найден, но есть CNAME — следует по цепочке CNAME (до 8 уровней).
  4. **Upstream-прокси** — если зона не локальная, запрос перенаправляется на вышестоящий DNS-сервер.
  5. **Кэширование результата** — ответ сохраняется в кэш с минимальным TTL из полученных записей.

- **`DNSCache`** ([`cache.go`](../internal/server/cache.go)) — потокобезопасный in-memory кэш с истечением по TTL. Ключ формируется как `domain|qtype`. Использует `sync.RWMutex` для конкурентного доступа.

### 5. Веб-сервер ([`internal/web/`](../internal/web))

- **`Server`** ([`server.go`](../internal/web/server.go)) — HTTP-сервер на чистом `net/http` (Go 1.22+ routing). Реализует:
  - REST API для управления зонами и записями
  - Раздачу статических файлов (CSS, JS) через `embed.FS`
  - SPA-роутинг: все пути, кроме `/api/*` и `/static/*`, отдают `index.html`

- **Шаблоны** ([`templates/index.html`](../internal/web/templates/index.html)) — единственный HTML-шаблон, SPA на ванильном JavaScript.
- **Статика** ([`static/app.js`](../internal/web/static/app.js), [`static/style.css`](../internal/web/static/style.css)) — клиентское приложение и стили (тёмная тема, GitHub-подобный дизайн).

### 6. Миграции ([`migrations/`](../migrations))

Две миграции:

1. **`000001`** — создание таблицы `zones` с уникальным индексом по имени зоны.
2. **`000002`** — создание таблицы `records` с внешним ключом на `zones(id)` и составным индексом `(domain, type)`.

## Схема базы данных

### Таблица `zones`

| Колонка | Тип    | Ограничения          | Описание              |
|---------|--------|----------------------|-----------------------|
| id      | INTEGER| PRIMARY KEY AUTOINCREMENT | Идентификатор зоны |
| name    | VARCHAR(63) | NOT NULL UNIQUE    | Имя зоны (FQDN, с точкой на конце) |
| ttl     | INTEGER| DEFAULT 300          | TTL по умолчанию для записей зоны |

### Таблица `records`

| Колонка | Тип         | Ограничения          | Описание                    |
|---------|-------------|----------------------|-----------------------------|
| id      | INTEGER     | PRIMARY KEY AUTOINCREMENT | Идентификатор записи    |
| zone_id | INTEGER     | NOT NULL, FOREIGN KEY → zones(id) | ID зоны          |
| domain  | VARCHAR(255)| NOT NULL             | Полное имя домена           |
| type    | VARCHAR(10) | NOT NULL             | Тип записи (A, AAAA, CNAME, MX, TXT, NS, SOA) |
| rdata   | TEXT        | NOT NULL             | Значение записи             |
| ttl     | INTEGER     | —                    | TTL (NULL = TTL зоны)       |

## Поток обработки DNS-запроса

```
Входящий DNS-запрос
        │
        ▼
  ┌─────────────┐
  │ Есть в кэше?│───Да──→ Вернуть кэшированный ответ
  └──────┬──────┘
         │ Нет
         ▼
  ┌─────────────┐
  │ Локальная   │───Да──→ Поиск в БД по domain + type
  │ зона?       │         │
  └──────┬──────┘         ├── Найдено → вернуть записи
         │                ├── Не найдено → ищем CNAME
         │ Нет            │       │
         ▼                │       ├── Найден → следуем по CNAME (до 8)
  ┌─────────────┐         │       └── Не найден → NODATA (пустой ответ)
  │ Upstream    │         │
  │ DNS-сервер  │         │
  └──────┬──────┘         │
         │                │
         ▼                │
  ┌─────────────┐         │
  │ Кэшируем    │◄────────┘
  │ результат   │
  └──────┬──────┘
         ▼
    Ответ клиенту
```

## Кэширование

- In-memory кэш с ключом `domain|qtype`.
- TTL записи определяется как минимальный TTL среди всех Resource Records в ответе.
- Для NXDOMAIN используется `MINIMUM` из SOA или 60 секунд по умолчанию (RFC 2308).
- Потокобезопасность обеспечивается через `sync.RWMutex`.
- Кэш можно отключить переменной `DNSO_CACHE=false`.

## Graceful Shutdown

При получении сигналов `SIGINT` или `SIGTERM`:

1. Останавливается DNS-сервер (завершаются активные соединения).
2. Закрывается HTTP-сервер.
3. Закрывается соединение с БД.
4. Программа завершается с сообщением в лог.
